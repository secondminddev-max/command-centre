Introduction — Why Autonomous Agent Systems

We are at an inflection point. Large language models have crossed a threshold where they can plan, execute, and self-correct well enough to run entire business workflows without human intervention. The question is no longer can AI do this? — it's how do you wire it together?

The core insight

A single AI assistant is a tool. A coordinated system of specialised AI agents is a business. The difference is architecture — not model capability.

The old model vs the new model

Traditional SaaS / freelance

  • You do the research manually
  • You write and schedule social posts
  • You compile reports in Excel
  • You invoice clients one by one
  • Bottleneck: your time and attention
  • Revenue ceiling: your working hours

AutonomousOps model

  • Researcher agent scans markets 24/7
  • GrowthAgent writes and posts autonomously
  • Report agent compiles HTML → PDF reports
  • StripePay agent handles checkout flows
  • Bottleneck: none — agents scale horizontally
  • Revenue ceiling: your imagination

What makes this architecture different

Most "AI automation" tutorials show you how to use a single prompt to do a single task. This blueprint shows you how to build a persistent, multi-agent organisation where agents have roles, relationships, memories, and specialisations — and where work flows automatically between them via a delegation chain.

Specialist Roles

Each agent owns a domain. The Researcher researches. The Designer designs. The StripePay agent handles payments. No agent does everything — specialisation creates quality.

Persistent State

Agents maintain logs, history, and context between runs. The system remembers what it did yesterday, what campaigns worked, and what to try next.

Event-Driven

Work triggers work. A new stock discovery triggers a report. A published report triggers a social post. A social post triggers a payment link. No manual handoffs.

What you'll be able to build after this blueprint

  1. A fully autonomous market research and reporting pipeline that produces sellable PDF reports
  2. An automated social media presence that posts research-backed content without manual effort
  3. A Stripe-integrated storefront where agents handle the entire checkout and delivery flow
  4. A real-time operations dashboard (the Command Centre) to monitor every agent's activity
  5. A policy and compliance layer so agents stay within safe operating boundaries
✅

Prerequisites

Basic Python or JavaScript, an Anthropic API key, and a Stripe account. The system runs locally or on any VPS. No proprietary cloud infrastructure required.

Agent Design Patterns — Roles, States & Behaviours

Each agent in the system follows a consistent design contract. This predictability is what makes the system composable — you can swap, upgrade, or add agents without rewiring everything else.

The Agent State Machine

Every agent cycles through four states. State transitions are logged and visible on the Command Centre dashboard in real-time.

IDLE
Waiting for work.

Agent is running but has no active task. Heartbeat is active. Will accept delegation immediately.

ACTIVE
Processing a task.

Agent is executing. Progress is streamed via SSE. Cannot accept new tasks until complete.

BUSY
Queued tasks.

Agent has additional work queued. Current task running; next task will begin on completion.

ERROR
Needs attention.

Task failed. Error logged. Reforger agent is notified for self-repair attempt.

Agent anatomy

# Every agent follows this contract
class BaseAgent:
  name: str          # unique ID
  role: str          # human-readable role
  state: AgentState  # IDLE|ACTIVE|BUSY|ERROR
  persona: str       # personality prompt
  skills: list[str]  # what it can do
  log: list[Entry]  # persistent history

  async def handle(task: Task) -> Result:
    # 1. validate against PolicyPro
    # 2. log to MetricsLog
    # 3. execute with LLM
    # 4. emit SSE progress
    # 5. return result
    ...

Persona system

Each agent has a persona prompt — a short paragraph that shapes its tone, decision-making style, and risk tolerance. The CEO is decisive and brief. The Researcher is thorough and citation-heavy. The Designer is opinionated about aesthetics. Personas create consistent, predictable behaviour across thousands of tasks.

ℹ️

Design rule: one agent, one job

Resist the urge to make agents do multiple things. A Researcher that also writes social posts becomes unpredictable. Specialisation is the source of reliability in a multi-agent system.

Delegation Architecture — CEO → Orchestrator → Specialists

The delegation chain is the nervous system of the entire operation. It defines how intent at the top level becomes executed work at the specialist level — without any human in the loop.

CEOSets objective
OrchestratorBreaks into tasks
SpecialistExecutes task
MetricsLogRecords result
CEOReceives summary

How a delegation is born

  1. CEO issues a high-level instruction: "Find undervalued ASX small-caps under $50M market cap."
  2. Orchestrator decomposes this into atomic tasks: screener query, data fetch, scoring, report generation.
  3. Each task is dispatched to the right specialist with a structured Task object containing context and constraints.
  4. Specialists execute and return results. Orchestrator aggregates and returns to CEO.
  5. CEO reviews the summary log and issues follow-up delegations if needed.

The Task object schema

// Task dispatched to any agent
{
  "id": "task_abc123",
  "from": "orchestrator",
  "to": "researcher",
  "priority": "high",
  "instruction": "Screen ASX
    small-caps under $50M
    market cap for value",
  "context": {
    "sector": "technology",
    "min_revenue": 1000000
  },
  "deadline": "2026-03-17T18:00Z",
  "callback": "orchestrator"
}

Delegation beams — visual feedback

The Command Centre renders active delegations as animated beams on the office floor canvas. When the CEO delegates to the Orchestrator, a bright arc appears. When the Orchestrator fans out tasks to five specialists simultaneously, five beams radiate outward. This gives you an instant visual picture of system activity — you can see at a glance whether the system is idle, running a single pipeline, or in full orchestration mode.

Delegation depth

The system supports 3 levels of delegation: CEO → Orchestrator → Specialist. Deeper nesting is possible but increases latency and error surface. Keep it flat.

Parallel dispatch

The Orchestrator can dispatch to multiple specialists simultaneously. A research job might fan out to Researcher + Scout + FinancialResearcher in parallel, then aggregate results.

Result aggregation

After parallel dispatch, the Orchestrator waits for all results, merges them into a unified context object, and returns a single structured summary to the CEO.

Error handling in the delegation chain

When a specialist fails, the error propagates upward. The Reforger agent intercepts ERROR states and attempts recovery via three strategies:

  1. Retry with context — re-run the same task with the error message prepended to the prompt
  2. Fallback agent — route to an alternative specialist with the same capability
  3. Escalate to CEO — if both fail, surface the failure as a priority notification via Telegram
⚠️

Circuit breaker pattern

Implement a circuit breaker on each agent. If an agent fails 3 times in 10 minutes, mark it as DOWN and route all new tasks to a fallback. This prevents cascade failures from taking down the entire system.

The social pipeline converts raw research outputs into platform-ready posts and publishes them without human intervention. The system currently posts to Bluesky via the AT Protocol, with the architecture ready to extend to X, LinkedIn, and Farcaster.

GrowthAgent prompt pattern

GrowthAgent receives a research summary and uses a structured prompt to generate platform-native content with CTAs:

"""
You are GrowthAgent. Given a research
summary, write a Bluesky post that:
- Leads with a compelling stat or finding
- Uses 2-3 relevant hashtags
- Ends with a clear CTA and link
- Stays under 300 characters
- Never sounds like an AI wrote it

Research: {summary}
Product URL: {checkout_url}
"""

SocialBridge dispatch

async def post_to_bluesky(content):
  agent = AtpAgent(
    service='https://bsky.social'
  )
  await agent.login(
    identifier=os.env.BSKY_HANDLE,
    password=os.env.BSKY_APP_PW
  )
  await agent.post({
    'text': content.text,
    'langs': ['en'],
    'createdAt': datetime.now()
  })
  MetricsLog.record('post_published', {
    'platform': 'bluesky',
    'content': content.text[:50]
  })

Curator quality gates

Before SocialBridge dispatches a post, the Curator agent runs four checks:

CheckCriteriaAction on fail
LengthUnder platform character limitTruncate and regenerate CTA
ToneNo AI-sounding phrases ("certainly", "I'd be happy to")Rewrite with GrowthAgent
ComplianceNo financial advice language (PolicyPro rules)Escalate to PolicyWriter
DedupNot identical to a post in the last 7 daysDiscard and re-trigger research

Campaign log and attribution

Every published post is logged in the Campaign Log with: timestamp, platform, post text, linked product URL, engagement metrics (fetched 24h later), and which research report triggered it. This creates a full attribution chain from research → post → revenue.

Data enrichment loop

Analyst scoring model

# Scoring weights (adjustable)
WEIGHTS = {
  "cash_runway_months": 0.20,
  "revenue_growth_yoy": 0.18,
  "director_buying":    0.15,
  "market_cap_usd":     0.12,
  "debt_to_equity":     0.12,
  "pe_ratio":           0.10,
  "sector_momentum":    0.08,
  "short_interest":     0.05,
}

def score_company(data):
  score = 0
  for metric, weight in WEIGHTS.items():
    score += normalise(data[metric]) * weight
  return round(score * 100, 1)

Stripe Checkout Integration

StripePay agent manages the full payment lifecycle — creating products, generating checkout links, handling webhooks, and triggering delivery. It turns a finished report into a revenue event within seconds of publication.

StripePay agent responsibilities

  1. Receive a product-ready signal with report path and metadata
  2. Create a Stripe Product and Price via API if not already existing
  3. Generate a Payment Link (shareable checkout URL, no server required)
  4. Pass checkout URL back to GrowthAgent for CTA embedding
  5. Listen for checkout.session.completed webhook
  6. On payment: trigger delivery agent to email report PDF to buyer
  7. Log revenue event to MetricsLog and notify CEO via Telegram

Creating a checkout link

import stripe

stripe.api_key = os.env['STRIPE_SK']

# Create product + price
product = stripe.Product.create(
  name="ASX Small-Cap Screener",
  description="Weekly report",
)
price = stripe.Price.create(
  product=product.id,
  unit_amount=4700,  # $47.00
  currency="aud",
)
link = stripe.PaymentLink.create(
  line_items=[{"price": price.id,
               "quantity": 1}],
)
return link.url

Telegram Alert Relay

The Telegram agent is the notification layer — it gives you a real-time window into the system's activity without you having to watch the dashboard.

When Telegram fires

  • New report generated and published
  • Payment received (with amount + product)
  • Agent enters ERROR state
  • PolicyPro blocks an action
  • Daily morning briefing summary
  • Unusual system metrics (CPU, API errors)

Telegram dispatch

async def notify(msg: str, level="info"):
  emoji = {
    "info": "ℹ️",
    "success": "✅",
    "warning": "⚠️",
    "revenue": "💰",
    "error": "🚨"
  }[level]
  bot = TelegramBot(os.env['TG_TOKEN'])
  await bot.send_message(
    chat_id=os.env['TG_CHAT_ID'],
    text=f"{emoji} {msg}"
  )

File Monitoring with FileWatch

FileWatch monitors the filesystem for changes and triggers agent workflows based on file events. It is the bridge between file-based outputs (reports, configs, data dumps) and the agent pipeline.

Watch rules example

      "trigger": "stripepay.create_product"
    },
    {
      "path": "data/raw/*.json",

Key use cases

  • New report in reports/ → auto-create Stripe product
  • New data file → trigger Analyst scoring
  • Config change → reload agent settings
  • Log file grows beyond threshold → alert SysMonitor
  • Credentials updated → restart dependent agents

Metrics & Observability — MetricsLog + SysMonitor

In an autonomous system, you need deep observability. Every agent action, every delegation, every external API call, every revenue event is logged. Without this, you're flying blind — you can't debug failures, prove ROI, or improve the system.

MetricsLog event schema

{
  "ts": "2026-03-17T09:42:11Z",
  "agent": "growthagent",
  "event": "post_published",
  "severity": "info",
  "duration_ms": 1243,
  "tokens_used": 892,
  "cost_usd": 0.0089,
  "metadata": {
    "platform": "bluesky",
    "post_id": "3kpq..."
  }
}

SysMonitor health checks

  • Agent heartbeat (30s poll — IDLE/ACTIVE/ERROR)
  • API key validity (Anthropic, Stripe, Telegram)
  • Disk space on reports directory
  • Memory and CPU for long-running processes
  • External service latency (Bluesky, Yahoo Finance)
  • Daily cost accumulation vs budget cap
⚠️

Set a cost cap

Always configure a daily token spend cap in SysMonitor. Runaway loops or prompt injection attacks can generate thousands of API calls. A $10/day cap prevents catastrophic billing events.

Daily briefing report

MetricsLog generates an HTML daily briefing at 7am each morning. It includes: agent uptime stats, tasks completed, API costs incurred, revenue generated, posts published, and any outstanding error states. The briefing is delivered to Telegram and saved to reports/daily_briefing.html.

Policy Enforcement — PolicyPro + PolicyWriter

PolicyPro is the compliance and safety layer. Before any agent executes an action that could be risky — posting to social, charging a card, sending an email — PolicyPro checks it against a ruleset. PolicyWriter lets you author and update those rules using natural language.

PolicyPro rule types

TypeExample
ContentNever post financial advice
Rate limitMax 3 posts per day
Spend limitMax $50 in Stripe charges/hour
Time windowNo posts between 10pm–7am
AllowlistOnly post to approved platforms
BlocklistNever mention competitor names

PolicyPro check flow

async def check(agent, action, payload):
  rules = await load_rules(agent)
  for rule in rules:
    result = rule.evaluate(action, payload)
    if result.blocked:
      MetricsLog.record(
        'policy_block',
        {'rule': rule.id, 'agent': agent}
      )
      Telegram.notify(
        f"🚫 PolicyPro blocked {agent}:
          {rule.description}",
        level="warning"
      )
      raise PolicyViolation(rule.id)
  return True
🔒

PolicyPro is your kill switch

If something goes wrong in an autonomous system, you need to be able to stop it instantly. PolicyPro's emergency halt rule disables all outbound actions across every agent with a single config change — no need to kill processes or SSH into servers.

Monetisation Patterns — How to Sell Reports & Blueprints

The system is built to generate revenue, not just automate work. Here are the proven monetisation patterns that work with this architecture — from one-time report sales to recurring subscription models.

Pattern 1: Report-as-Product

The screener pipeline generates a new report weekly. Each report is a standalone product with its own Stripe payment link. Price at $47–$97. Promote via GrowthAgent social posts. No subscription friction — buyers pay once per report.

Pattern 2: Research Subscription

Offer weekly report delivery as a $47/month subscription using Stripe Billing. Subscribers get every new report automatically emailed. Retention is high because each report is genuinely useful and fresh.

Example: 50 subscribers × $47/mo = $2,350 MRR on autopilot

Pattern 3: Blueprint Sales

Sell the knowledge of how to build this system. This document is an example. Price blueprints at $97–$197. The system writes them, the system sells them. Meta-recursion as a revenue model.

Example: This blueprint · $97 · 100 sales = $9,700 one-time

Expose the screener as an API. Charge per query or per monthly credit bundle. Developers and fund managers pay for programmatic access to your curated, AI-scored data pipeline.

Example: $0.50/query · 500 queries/day = $7,500/month

Scaling from 5 to 26 Agents

You don't build a 26-agent system on day one. You build 5 agents that work, then add specialists as demand reveals gaps. This chapter describes the progressive scaling journey — what to build when, and why.

Phase 1 — The Core 5 (Week 1)

CEO Orchestrator Researcher GrowthAgent StripePay

This is your minimum viable autonomous operation. It can research, create content, publish, and accept payments. Start here. Prove the model generates revenue before adding complexity.

Phase 2 — Operations Layer (Week 2–3)

SocialBridge Telegram Analyst Writer Designer FinancialResearcher

Add specialist agents to replace GrowthAgent doing too many jobs. SocialBridge separates distribution from content creation. Telegram adds visibility. Designer produces polished reports. Quality improves dramatically.

Phase 3 — Infrastructure Layer (Week 4–5)

MetricsLog SysMonitor PolicyPro PolicyWriter FileWatch Sentinel

At this point the system is generating revenue. Add observability and policy enforcement now — before a runaway agent causes a real problem. This layer makes the system trustworthy enough to leave running overnight.

Phase 4 — Full Operations (Week 6+)

Coder Tester Deployer Scout Curator Auditor Scheduler Navigator Builder Connector Reforger

At 26 agents, the system becomes genuinely self-sustaining. Reforger handles failures. Coder+Tester+Deployer can ship new features. Scout extends the research surface. The system now improves itself with minimal human intervention.

Anti-patterns to avoid

Anti-patternProblemFix
Adding agents before proving valueComplexity without ROIEach agent needs a measurable job
Skipping the infra layerSystem fails silently at scaleAdd MetricsLog before Phase 4
No policy layerAgents act without guardrailsPolicyPro before any financial agent
Single model for all agentsOver-spending on simple tasksHaiku for routing, Opus for strategy
