## CONCRETE IMPROVEMENT: Fix Watchdog False-Positive STALE Detection

### Problem
The Watchdog marks agents as STALE when their task string is unchanged for
~90-120 seconds, then forcibly restarts them. This is triggering false positives
on legitimately IDLE agents:

  - `researcher` and `coder` are idle BY DESIGN, waiting for task delegation.
    Their task string ("Idle -- awaiting X tasks") never changes because there's
    nothing to do — that's correct behaviour. Restarting them is disruptive and
    wastes resources.
  - `arxivscout` just finished a successful fetch and is sleeping between
    scheduled runs. Marking it STALE and restarting it mid-sleep breaks its
    fetch interval logic.
  - `autonomy` is intentionally inactive until manually triggered. Restarting
    it does nothing useful.

Evidence from logs:
  "DEAD agent [coder]: task='Idle -- awaiting coding tasks' reason=unchanged 182s"
  "DEAD agent [apipatcher]: task='Restarted' reason=unchanged 185s"

### Suggested Fix
Add an `idle` flag to agent state. Agents should call `set_agent(id, idle=True)`
when they enter a deliberate wait/sleep state. The Watchdog's STALE check should
skip agents where `idle=True`, or apply a much longer timeout (e.g. 10+ minutes)
for them.

Minimal implementation in agent_server.py:

  1. In set_agent(), accept an optional `idle` boolean field in the agent dict.
  2. In the Watchdog's STALE detection loop, skip any agent where agent["idle"] == True.
  3. In researcher, coder, apipatcher, arxivscout, autonomy — call
     set_agent(id, status="active", task="...", idle=True) when entering wait loops.

### Expected Outcome
- Eliminates spurious restart noise from Watchdog logs (currently ~40% of all
  log lines are false-positive STALE/DEAD/Restart entries for idle agents).
- Reduces unnecessary process churn on agents that are correctly dormant.
- Watchdog retains full crash-detection for genuinely stuck agents.
- Log signal-to-noise ratio improves, making real failures easier to spot.
